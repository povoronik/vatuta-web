Todo lo que suma a un negocio así

  • Botones, listas, formularios nativos
  • Plantillas por vertical listas para usar

🎯CRM y embudo

  • Historial, etiquetas, captura de leads desde la web
  • Ruteo de leads a la persona/sucursal correcta

📣Campañas / broadcast

  • Envíos masivos con plantillas de WhatsApp
  • Secuencias, recupero de carrito
  • Comment-to-DM en redes

📊Analytics y valor

  • Métricas de conversaciones, resolución, conversión
  • Reporte de valor mostrado al cliente (retención)

🔌Integraciones y comercio

  • E-commerce (Tiendanube/Shopify), Sheets, Zapier, API
  • Catálogo, pedidos y pago por chat
  • Logística / envíos

🏢Plataforma / negocio

  • Planes (contactos/conversaciones/agentes), free trial
  • Roles, seguridad, white-label, i18n
  • Ventaja local: español, pesos + Mercado Pago, factura AFIP, soporte acá

Lo que la matriz revela

Tenés el motor conversacional completo y hasta dos diferenciales que casi ningún competidor trae — presupuestos con PDF y logística de envíos — más cimientos de seguridad y tests de nivel empresa. Lo que falta es la capa de plataforma SaaS (multi-tenant, login, billing, alta self-service) y algunas features de UX moderna (flow builder visual, base de conocimiento, copilot). Es decir: te falta el "traje de SaaS", no el músculo.

04

Qué se le puede sacar al repo de PLUR

Auditoría a fondo de los 44 módulos del backend, frontend, base y tests. El veredicto en un número:

~45%del núcleo SaaS ya construido

Y es la mitad más difícil y valiosa: motor omnicanal con IA, bandeja, presupuestos/PDF, CRM, logística, seguridad y disciplina de tests. Todo single-tenant pero muy bien modularizado.

Un dato de oro para el multi-tenant: el código ya extrae phone_number_id y pageId de cada mensaje entrante — la pieza clave para enrutar por cliente ya está; solo falta la tabla que mapea ese id → tenant.

✓ Reusable · el núcleo

  • bot.js — orquestador omnicanal con IA (1.066 líneas)
  • gemini/agente — IA con function calling
  • whatsapp/messenger/meta — canales + firma HMAC
  • bandeja/conversacion/medios — inbox + media
  • leads — CRM + embudo
  • presupuesto·pdf, envios — motor (los diferenciales)
  • seguridad, push, email, tests, CI

◐ Específico de PLUR · generalizar

  • restauracion.js — máquina de estados de una vertical (527 líneas) → plantilla
  • ruteo.js — Clara/Ciudadela/Paso del Rey cableado → config por cliente
  • representantes/padron — MLM de PLUR → módulo opcional
  • Catálogo de restauración/Glasé → datos por tenant
  • Prompt con "plur.ar/glase.ar" → por tenant

✗ Falta para SaaS · construir

  • Multi-tenancy: tenant_id en cada tabla y query (hoy: 0 referencias)
  • Login propio: reemplazar Cloudflare Access por usuario/contraseña multi-cliente
  • Alta de canales por cliente: Embedded Signup + credenciales por tenant (hoy: 1 set global)
  • Billing + onboarding + super-admin: no existe nada
  • Generalizar lo específico de PLUR
🔑

El detalle técnico que abarata todo

Hoy todas las credenciales son globales del Worker (un WhatsApp, un Page, una app de Meta, en variables de entorno). Para multi-cliente hay que moverlas a una tabla tenant_credenciales en D1 y leerlas por request. Es trabajo acotado y mecánico — no una reescritura. Y permisos.js (deny-by-default) se reusa casi tal cual cambiando el eje "red de PLUR" por "tenant".

05

Propuesta para definir juntos

Con todo lo anterior, esta es mi propuesta de qué entra en la v1 (para salir a vender rápido apalancando lo que ya tenés) y qué queda para después. Es un punto de partida para que ajustemos antes de armar el plan.

Entra en la v1 (apalancás lo construido)

  • Bot omnicanal con IA (WhatsApp + IG + FB + web) multi-tenant
  • Bandeja unificada + handover
  • CRM de leads + embudo + captura web
  • Presupuestos con PDF (tu diferencial estrella)
  • Login propio + alta self-service + primer mes gratis
  • Billing con Mercado Pago · panel super-admin (tu tablero)
  • 1–2 plantillas de vertical para arrancar

Queda para v2+ (cuando haya clientes)

  • Flow builder visual (drag-and-drop)
  • IA con base de conocimiento por cliente + copilot
  • Campañas / broadcast masivo avanzado
  • Integración e-commerce (Tiendanube/Shopify)
  • White-label y multi-idioma
  • DB-por-tenant para clientes grandes

Las 4 decisiones que tenemos que cerrar antes del plan

Decisión 1 · Posicionamiento¿"Atención" o "ventas"?Whaticket vende atención multiagente; Leadsales vende "vendé más por WhatsApp". El mensaje cambia el producto. Mi recomendación: ventas (embudo + presupuestos es tu fuerte).
Decisión 2 · Diferencial central¿Cuál es tu "una cosa"?Tenés dos que casi nadie tiene: presupuestos con PDF y logística. Propongo apoyar todo el marketing en "el bot que cotiza y cierra la venta, no solo responde".
Decisión 3 · Alcance v1¿Salimos con el mínimo o con más?Mi recomendación: la v1 de arriba (motor + capa SaaS), sin flow builder visual todavía. Vender antes, no construir de más.
Decisión 4 · Verticales faro¿Con qué 1–2 rubros arrancamos?El producto es horizontal, pero conviene un mensaje afilado. ¿Los que ya conocés (venta directa, restauración/gastronomía), u otro?
